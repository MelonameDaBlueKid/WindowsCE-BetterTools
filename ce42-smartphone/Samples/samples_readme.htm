Smartphone 2003 SDK Samples

This document contains information on the samples included in this installation of the SDK. To run each of these samples, start eVC, and open the sample workspace (*.VCW) for the sample you want to examine or run.

 

By default all sample files get downloaded to \Storage\Application Data\Volatile\. To have eVC download samples to the Programs list change the following setting in eVC. With an open project select the "Project" menu item, subitem "Settings..." Select the "Debug" tab and change the "Download directory:" to "\Storage\Windows\Start Menu\Accessories". Remember to download directory is set per build configuration ("Win32 (WCE ARMV4) Debug" or "Win32 (WCE emulator) Release") and the sample needs to be rebuilt for the sample to appear in the Programs view. Programs only updates when the folder view changes, so you may need to navigate out of the accessories folder and back to see recently deployed applications.

 

Unless otherwise noted, all samples work on the emulator as well as the mobile device.

Win32 Samples

Autorun

  • Demonstrates an Autorun application for storage cards.
  • If you provide applications on storage cards, you must include an Autorun application on the card because the Smartphone doesn’t provide any way for the user to directly access the files system, and thus they wouldn’t be able to get to your application.

Back

  • Demonstrates how to support the system Back key in an application.
  • The Back key acts as a backspace key in edit boxes as well as a navigation tool to go back to the previous application. Correct use of the Back key in your applications is important to providing a good and consistent end user experience.

Cegetdev

  • Demonstrates getting information about the display capabilities of the device at runtime from the GetDeviceCaps function.

Cellcore – CeDialer

  • Demonstrates how to choose, open, and make a call on a TAPI line.
  • This sample also demonstrates how to write a common code base for Smartphone and Pocket PC.

Cellcore – ExTAPI

  • Demonstrates how to open a line and get general information.
  • This sample requires either a real device or a GSM radio attached to the emulator. The device or radio must have a SIM and service from a mobile operator.

Cellcore – ExTAPIAsync

  • Demonstrates how to use the asynchronous features of ExTAPI by trying to register an alternate operator.
  • This sample requires either a real device or a GSM radio attached to the emulator. The device or radio must have a SIM and service from a mobile operator.

Cellcore – SMS – Hello SMS

  • Demonstrates a simple SMS message sender.
  • This sample requires either a real device or a GSM radio attached to the emulator. The device or radio must have a SIM and service from a mobile operator that provides SMS.

Cellcore – TAPI – Hello TAPI

  • Demonstrates a Simple TAPI phone dialer. After selecting either dial option from the menu, the application will close and a blank Call Progress dialog will appear. Press the home button to return to the home screen.
  • This sample requires either a real device or a GSM radio attached to the emulator. The device or radio must have a SIM and service from a mobile operator.

Cellcore - WDP_SMS

  • Shows how to send and receive WDP messages over Short Message Service (SMS), using SMS to transport data.
  • This sample requires either a real device or a GSM radio attached to the emulator. The device or radio must have a SIM and service from a mobile operator that provides SMS.

CfgClient

  • Demonstrates how a push client receives a push message and sends a response via HTTP or WSP. 
  • This sample utilizes the device management API to process the configuration push message.

Closewindow

  • Demonstrates forcing a program into base state when it is re-launched. Make sure that child windows have a parent because this sample will not work if any window has a NULL parent.

CMTime

  • Demonstrates how to use connection manager by using it to establish a connection.
  • Connects to a time server and synchronizes the phone with the current time.
  • Near the start of the file cmtime.cpp (and identified by a comment block), the variable g_in_addr is initialized as the IP address for the time server the program should use to obtain the time. By default, g_in_addr is set to the IP address of a time server operated by the United States government.
    • If you are using the TimeServer application on your computer, you can change g_in_addr to the IP address of the computer running TimeServer.EXE.
    • If you want to use the sample to connect to a time server on the Internet, you can set the IP address for that server in g_in_addr. You can substitute the IP address of any time server that supports Time Protocol (RFC-868). Note that using a time server on the Internet requires that your computer be able to access port 37. Depending on your network and proxy server configuration, you may need to ask your network administrator or help desk what you need to do to access traffic on port 37. Generally, this will require you to install or configure a proxy client on your desktop computer. If your network is not allowing port 37 traffic through to your computer, you will see the message “…Finding Connection…” in the CMTime application window for about one minute followed by the message “Error getting time!”

CMTime – TimeServer

  • This is a Win32 console app that mimics a time server for use with the CMTime device sample. When you run TimeServer.exe the command window will indicate activity when an application connects to get the time.
  • An executable for this sample is included in the SDK. If you wish to build it yourself, you must do so using desktop Visual C++. This sample was compiled and tested using Visual C++ 6.0.

CreateCab

  • Demonstrates how to create, sign, and install a .cab archive file.

Currency

  • Demonstrate how to port a Pocket PC application (a simple currency converter) to Smartphone 2003.
  • A complete explanation of the way this sample is organized is contained in the readme.txt file in the Currency directory.

Customform

  • Demonstrates how to use the IMailRuleClient interface to intercept incoming MAPI messages.
  • Shows processing of incoming EMS messages.
  • Implements a custom read form to display EMS messages.

DeviceConfiguration

  • Demonstrates a device configuration web site for generating and configuring Smartphone devices.

DrawingText

  • Demonstrates simple Win32 text and shape drawing.

Enroll

  • Demonstrates how to use WinInet and security protocols to log in to secure web sites and download and store certificates.

Etcha

  • Shows how to use the navigation pad.
  • This sample is similar to the familiar drawing toy. It starts off in the center of the screen and draws as you use the navigation pad.

FileClient

  • Demonstrates a push client that receives a wireless push message, saves the message body to a file, and then calls ShellExecuteEx on that file.

GapiApp

  • Shows a well behaved Game API (GAPI) application on smartphone.
  • This sample specifically demonstrates handling WM_ACTIVATE messages, keyboard events for Home and Back and how to use the keyboard driver to locate input visualizations.
  • This sample is NOT an example of a well designed rendering system.  Specifically, the sample implements SetPixel as a function rather than as a Macro which significantly impacts performance.

GetUuid

  • Demonstrates how to access the device's unique ID. This ID can be used for enterprise asset management systems or other applications that require a device to be uniquely identified.

HelloPlug

  • Demonstrates a simple Home Screen plug-in for Smartphone.

HomeScreen

  • Demonstrates a more complex Home Screen plug-in for Smartphone.
  • Displays the device’ current memory state.

KeyTest

  • Displays WM_CHAR, KEY_UP and KEY_DOWN messages sent to the application.

MAPIRule

  • Demonstrates how to use the IMailRuleClient interface to intercept incoming MAPI messages.

MenuDemo

  • Demonstrates how to work with the soft keys and soft key menus.

MSMQ – VisAdm

  • The MSMQ Visual Administrator tool ported to function on Pocket PC. 
  • This sample also demonstrates how to make an application UI function on both Pocket PC and Smartphone devices.

MUIHello

  • Demonstrates how an application takes advantage of the Multilanguage User Interface (MUI) on Smartphone 2003. This is a very simple application that displays some strings.
  • Running this application on the device or emulator set to English causes the strings from the English resource file to be displayed, while switching to German causes the strings from the German resource file to be displayed. Note that the strings in this sample are not actually German words, just the English words followed by (Ger).

OwnerListView

  • Displays a simple owner-drawn ListView control.

PoomMaster

  • Demonstrates the use of the POOM API for managing and listing PIM contacts, tasks, and appointments objects.
  • This sample also utilizes an owner-drawn list box.
  • This sample code is a common code base for both Smartphone and Pocket PC devices.

PoomTasks

  • Implements a Tasks application by using the Pocket Outlook Object Model (POOM) API.
  • Data storage for the application is handled by POOM and is therefore available to any POOM-based application.

PSM

  • Demonstrates the use of persistent storage on Smartphone 2003 to store application data. All applications should be stored, along with any data they use because the RAM file store is very limited in size and anything not stored in the persistent storage is lost when the phone is turned off.
  • Since the emulator does not have a persistent file store, this sample functions, but behaves differently than it would on a real device.
  • This sample uses a #ifdef...#else construct to allow it to function if CSIDL_APPDATA is not defined. Pocket PC does not define CSIDL_APPDATA, so this is a good example for real applications to follow to ensure code portability between Smartphone and Pocket PC.

PViewCE

  • A Process/thread/module viewer.
  • This sample uses the ToolHelp API (tlhelp32.h). Applications using the ToolHelp API are required to be signed using a Privileged certificate in order to function correctly.

RAPI – Pdblist

  • Uses the Remote API to list databases on the device.
  • This sample is a desktop application.

RAPI – Pdir

  • Uses the Remote API to list files in a given directory on the device.
  • This sample is a desktop application.

RAPI – Pget

  • Uses the Remote API to copy a file from the device onto the desktop machine it is connected to.
  • This sample is a desktop application.

RAPI – Pput

  • Uses the Remote API to copy a file to the device from the desktop machine it is connected to.
  • This sample is a desktop application.

RAPI – PregDump

  • Uses the Remote API to demonstrate how to get a list of registry entries from the connected device.
  • This sample is a desktop application.

RAPI – Prun

  • Uses the Remote API to demonstrate how to execute an executable file on the connected device.
  • This sample is a desktop application.

RAPI – Ptree

  • Uses the Remote API to demonstrate how to list all the files and directories on the connected device.
  • This sample is a desktop application.

RegDemo

  • Shows how to get and set registry values.
  • This sample provides a pair of edit boxes and allows you to view and set the two pre-determined registry values. If you set the values, quit the program and restart it, you will see that the values were persisted.

SendMail

  • Shows programmatic sending of an e-mail message using the CEMAPI messaging API.
  • This sample has no user interface, it simply puts a trivial message (with attachment) in the outbox of each installed account.

SetupDLL

  • Demonstrates how to implement a custom setup DLL for custom application installation, which is installed via cab file.
  • This sample is also used with the CfgClient sample.

ShowFontInfo

  • Demonstrates font rendering on the device and allows adjustments to font parameters. 
  • This sample also demonstrates a single-compilation application that runs on both Pocket PC and Smartphone.

SIMMgr

  • Demonstrates how to read the phone book entries off the SIM.
  • This sample requires either a real device or a GSM radio attached to the emulator. The device or radio must have a valid SIM.

SystemInfo

  • Displays information about the device or emulator. This includes information about processor, OS version, power status, system time, etc.

SystemParameters

  • Shows how to retrieve various system parameters.

VirtualListView

  • Demonstrates using a virtual list view.

Winmine

  • Demonstrates drawing of bitmaps and handling user input.
  • This sample is an implementation of the popular Minesweeper game.

WMHibernate

  • Demonstrates memory management features by allocating memory using malloc until the automatic memory management features take effect.
  • This sample allows you to keep malloc'ing memory until the OS sends the WM_HIBERNATE message, at which point a MessageBox pops up, warning you of low memory.  After dismissing the message box the sample app closes in response to the WM_HIBERNATE message.
  • This sample is designed to force a WM_HIBERNATE message from the OS. Memory is allocated by malloc (heap allocation) but no corresponding calls to free are made. The sample relies on the OS cleaning up all allocations when the process shuts down. Therefore, this sample is not to be used as an example of good programming practices.

pIEHelper
