Sample Overview

This sample demonstrates the process of configuring a device from a web site, using various optional transport methods (SMS, ActiveSync, or even e-mail). The sample consists of several components, including the web site itself (ASP), the control which configures the device (ActiveX, hosted on the aforementioned web site), a deferred installer distributed with the ActiveX control, and the configuration XML generated by the control. The entire sample exists only on the desktop and is meant to configure the device, rather than actually run on it. This sample will be most useful to those wishing to set up a corporate or operator website which will deliver a consistent configuration to multiple devices.

 

The sample files are separated into main directories, Website and DeviceConfigCtl. The Website folder contains the files relevant for the device configuration website, and contains various options for configuring the device. The website depends on an ActiveX control called DeviceConfigCtl for one of the four configuration delivery options. The source for the ActiveX control can be found in the aforementioned DeviceConfigCtl folder. Implemented using ATL, this control provides the functionality of installing a CPF file on to the device using ActiveSync. The ActiveX control in turn depends on an executable named DeferredInstaller.exe which, as the name suggests, is used to send configuration to a device at a later time if no device is connected at the time of configuration. The source for DeferredInstaller.exe is included in the DeferredInstaller subfolder under the DeviceConfigCtl folder.

 

 

Requirements and Dependencies

The website requires itself to be hosted on a Windows Server or a Windows machine running Microsoft Internet Information Server (IIS). The site also requires that a perl scripting engine be installed on the server so that the perl scripts that are part of the site can be run with adequate permissions. In addition to these, the following requirements also apply depending on the extent of functionality intended for the website.

 

  • In order to use the “Send Via Email” feature, the server machine should have either CDONTS or the Microsoft .NET Framework installed.
  • In order to use the “Send Via SMS” feature, the server machine should have access to a Over-the-Air (OTA) push server.
  • In order to use the “Send Via ActiveSync” feature, the ActiveX control (deviceconfigctl.cab) should either be signed with a valid Authenticode certificate and marked “Safe for Scripting and Initializing” or the users of the site should be notified to change their client browser settings accordingly.
  • If the user of the website wants to configure locked down Smartphones, a secure code signing mechanism should be in place such that the CPF (Cab Provisioning Format) file generated by the perl scripts in the website can be signed by the appropriate Authenticode certificates that are already provisioned on the device. The perl scripts in the Website folder have placeholders in them where scripting code for the same could be added.

 

The ActiveSync CPF Installer control (DeviceConfigCtl) source code in the DeviceConfigCtl folder is an ATL 7.0 project, built using Visual Studio .Net. The source for DeferredInstaller.exe, which is in the DeferredInstaller folder, is a Win32 project, also built using Visual Studio .Net. In order to rebuild these if changes are made to sources, Visual Studio .Net should be installed. The ActiveX control and the Deferred Installer are already pre-built and packaged into the DeviceConfigCtl.cab file in the Website folder. So if no modifications are required, there is no need build either of these projects. Building a Release version of the DeviceConfigCtl project requires that the Cabarc.exe tool, which is part of the Microsoft Cabinet SDK, is installed on the development machine. See the “Building the Sample” section for more details on how to rebuild the Visual Studio .Net projects.

 

Topics

The topics covered by this sample are the following:

 

  • Generating XML configuration files for configuring Smartphone and Pocket PC devices.
  • Different ways to send configuration settings to Smartphone and Pocket PC devices.
  • Use of Remote API (RAPI) and CeUtils provided by ActiveSync to interact with Smartphone and Pocket PC devices.
  • Building desktop ActiveX controls to interact with Smartphone and Pocket PC devices.

 

Building the Sample

The Website does not require any building step. It only requires a few setup steps to get it up and running. See the “Running the Sample” section for setup instructions for the website.

 

As mentioned earlier, the DeviceConfigCtl and DeferredInstaller projects are Visual Studio .Net projects. Double click on the <projectname>.sln file in the respective directory to open the project in Visual Studio .Net. Before any of the projects can be built, Visual Studio should be pointed correctly to locate the files it needs to build. The following steps should be taken for this:

 

  1. Open the Options dialog by clicking Tools->Options from the main menu in Visual Studio .Net IDE.
  2. Select “VC++ Directories” in the “Projects” sub-folder.
    1. Select “Include Files” in the “Show directories for:” drop down box. Add “<SDK_INSTALL_DIR>\ActiveSync\Inc”. SDK_INSTALL_DIR is the installed path of this SDK.
    2. Select “Library Files” in the “Show directories for:” drop down box and add “<SDK_INSTALL_DIR>\ActiveSync\Lib”
    3. Select “Executable Files” in the “Show directories for:” drop down box and add the location where Cabarc.exe exists.

 

 

The DeferredInstaller project should be built first prior to building the DeviceConfigCtl project if any modifications are made to the Deferred Installer source files. This can be automated by adding the DeferredInstaller project as a dependency or a pre-build step in the DeviceConfigCtl project. After building the DeferredInstaller project, the EXE file should be copied from its release directory to the DeviceConfigCtl directory.

 

Building the DeviceConfigCtl project is simple. Double click on the Visual Studio solution file (.sln), make the required changes to the source files if any, and build the solution. This will generate the DeviceConfigCtl.dll file and will register the control on the development machine for debugging purposes. If the Active Configuration of the project is set to “Release”, there is an additional custom post-build step for generating the CAB file for the ActiveX control. The post-build step uses the DeviceConfigCtl.inf file in the same folder and calls Cabarc.exe to package up the Control’s DLL and DeferredInstaller.EXE into a CAB that can be deployed in the Website. If any changes are made to the control, bump up the version number of the DLL before building the solution. This can be done by opening the version strings in the resource view of the project. The version number should also be updated in the INF file and in ASInstall.asp file in the Website.

 

Running the Sample

 

The following steps need to be taken when the setting up the website for the first time.

 

  1. Copy the entire content from the Website folder to the wwwroot folder or a different virtual folder under IIS.
  2. Setup IIS permissions as required for the intended user group.
  3. Give write permission for the “Profiles” directory to the intended user group.
  4. Give execute permission for the perl scripting engine to the intended user group.
  5. Edit “global.asa” file in the root folder of the website and make sure that the “Loginpage” variable points to the correct path. This path depends on how the website is configured.
  6. Edit “Admin_Email.inc” file in the “inc” folder in the website with the correct email address. This will the “From” address for configuration emails sent from the site, when using the “Send Via Email” option.
  7. Edit “SMTP_Server.inc” file in the same folder with the IP address of the SMTP server which will be used for sending emails.
  8. Edit “TPPG_address.inc” file with the URL for the OTA push server. This is required to use the “Send via SMS” option of delivering the configuration to the device
  9. Edit “TPPG_name.inc” file with the friendly name of the OTA push server.

 

Notes

The ActiveX control in this sample is not marked safe for scripting and initialization. Hence it might not work properly on some client browsers, even if the control is signed with a valid Authenticode signature. The problem could be fixed by either changing the security settings of client browser accordingly or by including code in the control’s source files to mark the control safe for scripting and initializing. Please follow instructions and guidelines given in MSDN to mark an ActiveX control safe for scripting and initializing.
